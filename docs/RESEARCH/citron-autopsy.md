# citron-script 尸检报告：AI 剧本为什么「拖沓、无高潮、割裂、无伏笔」

> 分析对象：https://github.com/Citronetic/citron-script（浅克隆于 2026-07-09，HEAD=dfe7bf3）
> 本地副本：`/private/tmp/claude-501/-Users-shuai-workspace-jinko-dev-loop/b6ceb54c-67f8-46d1-8904-6369ddc7065d/scratchpad/citron-script`
> 结论先行：**它的病不在「不懂编剧方法论」——方法论文件反而是全仓库最好的资产；病在「规划层与执行层之间没有任何机制保证规划被执行」**：剧本这道观众直接消费的工序恰恰是全流水线唯一没有审计、没有状态传递、用最便宜模型批量跑出来的一环。

---

## 0. 项目全貌

citron-script 是「小说 → 竖屏短剧剧本」的工作台：单进程 Node/Express + SQLite，同时暴露 REST（界面）与 MCP（本地 Claude 当"大脑"）。仓库内含 6 部作品共 251 集成品存档（每集 90 秒 / ≈225 字台词）。

关键目录：

| 路径 | 内容 |
|---|---|
| `data/skills/script_*.md` | 编剧方法论 7 份（决策/事件/骨架/改编/剧本/监督/源小说填充） |
| `data/skills/story_skills/` `art_skills/` | 14 个题材包 + 13 个画风包 |
| `src/mcp/tools.ts` | 12 个 MCP 工具（读章节/事件/骨架/剧本，写骨架/策略/剧本） |
| `src/agent/`（stages.ts / runtime.ts / pipeline.ts） | 服务端 Agent 运行时：stage 化推理 + 确定性审计 + 带反馈重试 |
| `src/gen/*Audit.ts`（约 15 个） | **全部是制作端（分镜/资产/视频提示词）审计，无一个剧本叙事审计** |
| `scripts/regenerate-scripts-openrouter.mjs` | 251 集存档的批量（重）生成脚本，默认模型 `openai/gpt-4o-mini` |
| `.claude/workflows/zest-audit-loop.js` | 多 Agent 审稿回环（仅 EP01 试点，未用于存档生产） |
| `NN-项目/`（章节事件.md / 故事骨架.md / 改编策略.md / scripts/EPxx.md） | 6 部作品的中间产物与成片剧本 |

数据模型只有四张核心表：`project / chapter / plan(骨架+策略) / script`。**没有任何一张表表示节拍、伏笔、角色状态或跨集线索。**

## a. 生成管线是什么

实际上并存**三条**生成路径（这本身就是问题之一，见 §b-3）：

**路径 1：本地 Claude 经 MCP 串行扮演五角色**（README 宣传的正统路径）
`决策 → 事件提取 → 故事骨架 → 改编策略 → 逐集剧本 → 监督`。有大纲层（骨架含逐集表格：集标题/章节范围/戏剧功能/集末钩子/付费点），逐集生成、每次一集、单轮上限 5 集。上下文传递：剧本 skill 规定 `get_script_content(ids) 只允许获取最后一集剧本内容`——即只能带上一集全文，没有更早剧集、没有任何状态摘要。**决策层 skill 明文规定「阶段3（剧本编写）不需要监督层审核」**（`script_agent_decision.md` L142）。

**路径 2：服务端 Agent 运行时**（`src/agent/stages.ts`）
每 stage 一次 OpenRouter 调用 + 写库 + 确定性审计 + 阻断重试（最多 3 次）。但看 `script` stage 的 `buildContext`（L259-274）：上下文 = 项目信息 + 目标集名 + 改编策略 + 骨架 + 章节事件图谱。**没有上一集剧本、没有相邻集内容**；且 `script` stage 是所有 stage 中**唯一没有 `audit` 函数的生成步骤**——分镜表、导演规划、衍生资产、分镜面板、视频提示词全都有审计重试闭环，唯独剧本没有。

**路径 3：批量重生成脚本**（`scripts/regenerate-scripts-openrouter.mjs`）——仓库里 251 集存档的实际主要来源
- 默认模型 `openai/gpt-4o-mini`（L9），temperature 0.72，单集 max_tokens 3400；
- **一次 prompt 批量生成 5 集**（`--batch-size` 默认 5）；
- 批内只有第 1 集拿到上一集结尾（截尾 1600-2200 字），**第 2-5 集的 brief 里 `prevScript` 传空串**（L354：`prevScript: index === 0 ? prevScript : ""`），且空串时模板文案是「**这是第一集，无上一集**；若本批前一集已生成，请顺序承接前一集钩子」——即每批 5 集中有 4 集被明确告知"你是第一集"；
- 质量门 `validScript()`（L214-216）只查格式：长度≥1000 字符、有标题、有「## 剧情梗概」、含 △、≥2 个 `---`。**没有任何叙事断言**；
- 重试一次仍不合格时打个 `warn-EP..-format` 就照样写盘落库（L543-544）。

**结论**：有大纲层（且质量不错），是分集生成，但上下文传递薄、不一致、且在最常用的批量路径上接近于无。

## b. 四个症状的根因定位

### b-1. 「拖沓」——缺的不是"最小推进约束"的文字，而是它的可验证性

骨架 skill 里其实写了极狠的反注水条款（「一集一进（不可逆）」「这一集删掉剧情断不断」自检、每集 2-3 节拍+1 反转），剧本 skill 也重复了一遍。**但这些约束只存在于 prompt 形容词层**：

1. `validScript()` 验的是 △ 和 `---` 的个数，不验"本集是否有不可逆推进"；
2. 剧本 skill 的「自查清单」明文标注「仅供内部校验，**不输出**到剧本中」——一个既不可观察、也没人复核的自查，等于没有自查；
3. 弱模型在 3400 token 预算内用重复注水填满篇幅。实证：`02-my-miracle-luna/scripts/EP08.md` 共 12 场戏，从 1-5 到 1-12 反复重演「Ash 震惊 / Rylee 坚定 / Kano 怒吼」同一个情绪节拍，甚至出现张冠李戴（骨架规定 Rylee 说「疼吗？」，成片写成无名「声音：（低沉）疼吗？」）；
4. **串集重复放大了剧级拖沓**：EP06 的梗概已经把骨架分给 EP07（念拒绝词）和 EP08（亮身份反转）的节拍全部演完，EP08 又原样重演一遍——观众连看三集同一房间同一事件，主线原地踏步。根因是每集生成时只拿到"本集那一行"，**没有「相邻集写什么、本集禁写什么」的负向边界**，也没有生成后与邻集对读的查重。

### b-2. 「没有情节高潮」——规划了高潮，但没有任何工序保证高潮落地

骨架层的高潮规划其实很完整：付费卡点按 10/30/50/70/90% 分布、「强卡点 vs 弱卡点」对照表、「戛然而止在最高燃点」铁律、监督层还有专门的付费点分布审核维度。断裂发生在两处：

1. **监督层只审骨架和策略，不审剧本**。`script_agent_supervision.md` 的任务识别表里只有「骨架审核 / 策略审核」两类；决策层明文「阶段3 不需要监督层审核」。也就是说：卡点在骨架里设计得再好，剧本写没写出来，**没有任何角色负责验收**。
2. **高潮被提前泄洪**。EP06 提前引爆了 EP08 的「当众亮血统」反转（见 b-1-4），等观众看到规划中的卡点集 EP08 时，"高潮"已是二手信息。卡点集与普通集走完全相同的流水线、相同的模型、相同的 token 预算，没有任何"高价值集专项打磨"工序。
3. 批量脚本的 prompt 虽然喊了「付费点集要更狠」，但既不知道全季情绪曲线的当前位置，也没有"这是全季最高点"的比较基准——90 秒×每集都要求「≥1 反转+悬崖结尾」的均匀轰炸，反而把曲线拉平：**当每一集都是"高潮"时，就没有高潮**。

### b-3. 「剧集之间没有联系、割裂」——上下文在三个层面同时丢失

1. **批内丢失**：批量模式下每批 4/5 的集数被告知「这是第一集，无上一集」（见 §a 路径 3）。这是最直接、最实锤的割裂来源。
2. **架构性丢失**：服务端 `script` stage 的上下文根本不含上一集；MCP 路径也只允许带"最后一集"，无角色状态、无关系进展、无未回收线头的任何摘要。上一集尾巴 1600 字只能保证"接住钩子"，接不住人物弧、情绪曲线和已揭露信息集合（谁已经知道什么秘密——信息差是短剧核心机制，却没有任何状态追踪）。
3. **来源混杂丢失**：同一季里不同集来自不同生成批次/模型/方法论版本。实证：EP05 文件头风格标签是「狼人虐恋·灰姑娘逆袭+命定救赎」且场景丰满（≈3.6KB、△描述细腻），EP06 标签是批量脚本 `PROJECTS` 表里的「拒绝配偶·灰姑娘逆袭+隐藏血统+命定救赎」且干瘪（2.5KB、每场 2-3 行）——相邻两集笔触、密度、连贯性肉眼可辨地断层。**没有任何"同季产物必须同源同版"的治理机制**，重生成单集不触发邻集重审。

### b-4. 「没有伏笔」——伏笔在整个系统里没有表示、没有追踪、没有验收

全仓库 grep「伏笔」：只出现在 skill 的顺带建议里（「为后续埋伏笔」「关键物证/伏笔要多次提醒」「伏笔镜头不加标注」）。**结构层面为零**：

- 四张表没有伏笔/线索实体；骨架逐集表有「集末钩子」列（下一集拉力），但没有「本集埋什么、第几集回收」列；
- 章节事件提取 schema（`script_execution_events.md`）捕捉「钩子/关键锚点」，但都是章内属性，不构成跨集 setup→payoff 配对；
- 监督层审核维度、zest 回环三视角评委、`validScript()`——没有任何一处检查「到期未回收的伏笔」或「无铺垫的突兀回收」；
- 每集生成只看见自己那一行骨架 + 本集对应章节事件，模型**在结构上不可能**为 30 集后的爆点埋线，也不知道 3 集前埋过什么可以回收。伏笔只能靠原著自带 + 运气残留。

## c. 有没有审查/修订环节？有没有评分反馈闭环？

**有，但覆盖面恰好反着放。**

| 环节 | 存在？ | 覆盖 | 效力 |
|---|---|---|---|
| 监督层 Agent（A/B/C/D 评分 + 问题清单） | 有 | **只审骨架、改编策略**；剧本明文豁免 | advisory——「只提建议不擅自改，改不改由人定」 |
| 服务端确定性审计 + 阻断重试 | 有，做得好 | 分镜表/导演规划/衍生资产/分镜面板/视频提示词（约 15 个 audit 模块，含视觉连续性审计 `continuityAudit.ts`） | gate——medium/high 阻断，带反馈自动重写，最多 3 次 |
| 剧本叙事审计 | **无** | `script` stage 无 audit 函数；批量脚本只有格式 `validScript` | — |
| zest-audit-loop（3 视角评委并行审 + 自动打回重写 + 收敛落库） | 有，真实跑通 | 单集范围；三视角=红线情绪钩子/忠实人设/台词格式，**无跨集连续性视角、无季级视角** | gate，但只做过 EP01 试点（落库名「EP01·回环试点」），**从未用于 251 集存档的生产**；且零瑕疵档被证实会震荡不收敛 |
| 评分/教训沉淀闭环 | 制作端有（`docs/production-lessons/` + `productionLessons.ts` + `memories` 表带 embedding 召回） | **剧本端没有等价物**：A/B/C/D 打完即弃，`agentRuns.auditLevel` 只记账不归纳，没有"叙事教训写回方法论"的通道 | — |

这是全项目最刺眼的不对称：**工程投入全押在"画面别穿帮"上（资产 id 合法性、视觉连续性、提示词合规），而"故事好不好看"这个用户实际抱怨的维度，一行验收代码都没有。** 团队甚至已经造好了正确的机器（runtime 的 audit-retry 闭环、zest 回环、memories 召回），只是没有把剧本叙事接进去。

## d. Prompt 层面的问题

与直觉相反，prompt **并非只写了"写什么"**——「节拍怎么走」写得相当具体（开篇 3 秒入冲突、每集≥1 不可逆推进+≥2 情绪点+≥1 反转、集末停在可见事件前一秒、单句≤20 字、情绪公式 1 核心+1 辅助+1 钩子）。真正的问题是五个更隐蔽的层面：

1. **节拍约束停留在形容词，没有结构化的中间产物**。骨架（战略层）与剧本（成稿层）之间缺一层"本集节拍单"（beat sheet）：场序 → 每场功能 → 反转位置 → 钩子内容。模型从"一行表格"直接跳到 1500 字成稿，节拍走不走样全凭模型自觉。
2. **正向指令满溢，负向边界缺失**。prompt 从不告诉模型「下一集会写什么，所以本集不许写到哪」——串集泄洪（EP06 吃掉 EP08）在 prompt 结构上就无法避免。`stages.ts` ironLaws 有一句「不串集、不越集」，但既没给邻集内容作参照，也没有事后查重。
3. **自查被要求"心里过一遍、不要输出"**——不可观察即不可执行，尤其对 gpt-4o-mini 这个量级的模型。
4. **可验证的格式锚点（XML 标签、标题三行、△、---）与不可验证的叙事要求混在同一张清单里**，而校验器只测前者——模型被训练信号引导为"格式合规优先"。
5. **批量 brief 含有主动有害的指令**（「这是第一集，无上一集」发给 EP07），以及「旧版本集仅作素材不要照抄」同时把旧稿全文塞进上下文——弱模型必然锚定旧稿，重生成退化为轻度改写。

另有一个非 prompt 但同层的问题：**创作核心工序用了全链路最便宜的模型**（gpt-4o-mini 批量跑 251 集），而方法论 skill 是按 Claude 级理解力写的（大量隐喻式行话：「压得越狠反弹越爽」「戛然而止在最高燃点」）。指令复杂度与执行模型能力严重错配。

## e. 值得保留的资产

1. **短剧方法论知识库**（`data/skills/script_execution_skeleton.md`、`script_execution_script.md`）：付费卡点 10/30/50/70/90% 比例体系、强/弱卡点对照表、五大类型节奏框架、情绪基调占比、信息差三型 taxonomy、钩子四型 taxonomy、爆/虐/爽三点定义与「爽点=装+打脸+震惊+收获」公式、三个可套用情绪模板、台词规范（≤20字/遮名可辨/戳软肋）。**这是领域知识密度极高的成品，直接可复用。**
2. **剧本格式规范**：文件头三行 + 剧情梗概 + `场号 场景名 时/光` + △描述（写"人怎么干"可直供 AI 视频）+ OS/V.S. + `[硬切]/[闪回]/[叠化]` 转场 + 竖屏构图约束 + 150字/分钟时长换算。已被下游分镜/出图链路验证可消费。
3. **章节事件图谱模式**（`script_execution_events.md` + 各项目 `章节事件.md`）：「梗概/关键节拍/人物/转折钩子/改编价值」五段式，把几十万字原著压成可寻址事件地图——中间表示设计得很好，样例质量高。
4. **监督 rubric**（`script_agent_supervision.md`）：A/B/C/D 评分标准（按严重/中等问题数定档）、短剧通用红线清单、跨阶段一致性检查（删减决策对齐、章节全覆盖）——直接可移植为审稿 Agent 的评分卡。
5. **zest-audit-loop 编排骨架**（`.claude/workflows/zest-audit-loop.js` + `docs/AGENT-LOOP.md`）：prep→author→3 视角评委并行→闸门→带合并意见重写→收敛才落库；两档收敛判据（可用档/零瑕疵档）与「未收敛=不落库」安全属性；试点还留下了宝贵教训——零瑕疵档会在"删OS↔字数不足"间震荡，收敛判据必须防对偶目标拉锯。
6. **服务端 stage/audit/retry 运行时模式**（`src/agent/runtime.ts`、`stages.ts`、`pipeline.ts`）：声明式 stage（buildContext/write/audit）+ 确定性审计阻断重试 + 门禁式流水线状态机——机制正确，只欠叙事侧的 audit 实现。
7. **题材/画风包**：14 个 story_skills（题材叙事手法，如悬疑包的「伏笔镜头不加标注」防 AI 感条款）+ 13 个 art_skills + 视频模型 prompt 模板。
8. **6 部 251 集存档本身**：作为新系统的评测语料（好坏样本都有——EP01/EP05/EP16 与 EP06/EP08 形成天然对照组）与回归测试基线。

## f. 新系统必须避免的 10 条教训（机制级对策）

1. **教训：规划与成稿之间没有「节拍契约」，骨架写得再好也约束不了成稿。**
   对策：在骨架与剧本之间增设**每集节拍单**为一等工件（本集必须发生的不可逆事件 / 场序与各场功能 / 反转位置 / 集末钩子原文 / **本集禁写清单**=邻集节拍）；写手角色的输入是节拍单而非"骨架里你自己找"，验收角色逐项核对节拍单落地情况——契约不满足即打回，属流程门禁而非 prompt 措辞。

2. **教训：跨集上下文=「上一集尾巴 2000 字」，角色状态、关系进展、信息差全靠模型脑补。**
   对策：建立**故事状态账本**（story bible / state ledger）作为持续更新的一等文档：角色当前处境与目标、关系矩阵、"谁知道什么秘密"的信息差表、未回收线头清单；每集开工前必读、交稿后由专责角色增量更新并作为下一集的强制输入。状态账本的更新本身要过审（漏记=事故）。

3. **教训：伏笔没有表示、没有追踪、没有验收——系统结构上不可能产出伏笔。**
   对策：建立**伏笔台账**（setup→reminder→payoff 三态记录：埋设集/提醒集/回收集/当前状态），在季规划阶段就登记主要伏笔；审稿环节跑两条确定性检查：「到期未回收」与「未埋先收」，任一命中即阻断。回收集的验收标准里必须包含"回指埋设场景"。

4. **教训：质量门=格式门。`validScript` 只数 △ 和 `---`，叙事崩坏照样落库。**
   对策：验收清单必须包含**可判定的叙事断言**，每条对应一个已知症状：本集有无不可逆推进（对照节拍单）、集末是否可见事件悬崖（引用原文判定）、是否重复上一集节拍（相邻集对读查重）、是否提前泄洪后续卡点（对照伏笔台账与节拍单）。判不出来的形容词不许进验收标准。

5. **教训：审查覆盖倒置——规划有审、剧本免审；监督只有建议权没有否决权。**
   对策：审查资源按"离观众距离"分配：**成稿剧本必须过独立审稿角色的阻断式门禁**（advisory→gate），并分三层：单集审（红线/格式/节拍）、**跨集审（与前后集对读连续性）**、季级审（高潮曲线与伏笔回收复盘，每 N 集跑一次）。「未收敛=不落库」作为不可豁免的安全属性（zest 回环已验证此属性可行）。

6. **教训：为省钱把 5 集塞进一次调用并行生成，批内 4/5 的集数被告知"这是第一集"——生成拓扑践踏了叙事依赖链。**
   对策：流程层面规定**集 N 开工的前置条件是集 N-1 已通过验收**（串行链式 + 状态账本传递）；并行只允许发生在无叙事依赖处（不同项目、同集多稿竞争）。工单系统用依赖关系显式建模，而不是靠脚本循环里的约定。

7. **教训：同一季混杂多条管线、多个模型、多个方法论版本的产物，无版本治理；重生成单集不触发邻集复核。**
   对策：每个产物落库时携带**生成配置指纹**（模型/skill 版本/上游工件哈希）；季内指纹不一致即审计告警；任何单集重生成自动开出「邻集连续性复核」工单。创作核心工序的模型选型作为受治理的配置项，禁止默认降级到最便宜档。

8. **教训：卡点集/幕末集与普通集走同一条流水线、同一预算——规划中的高潮在执行层被均质化磨平。**
   对策：高价值集（卡点/幕末/季终）在工单系统中**单列工种**：更强模型或多稿竞争、专用 rubric（戛然而止点位置、公开大场合、不可逆巨变、铺垫链完整性）、强制人工或高配 Agent 终审；同时季级审负责压制普通集的"伪高潮"，保证情绪曲线有落差——高潮是全季比较级，不是单集绝对值。

9. **教训：评分打完即弃，剧本端没有教训沉淀闭环（制作端反而有 production-lessons + memories 召回）。**
   对策：复制制作端已验证的模式到叙事端：审稿发现的**系统性**问题（同类问题出现≥N 次）由 reflect 类角色归纳进「叙事教训库」，教训库注入后续写手/审稿的上下文，并定期提案修订方法论文档本身——评分数据（A/B/C/D 分布、打回原因分类）作为流程健康度指标可观测。

10. **教训：把自查设计成「心里过一遍、不要输出」——不可观察的约束等于不存在，尤其对低配模型。**
    对策：一切自检改为**显式产物或外部校验**：写手随稿提交结构化自检表（逐项断言+证据引用），或由确定性校验器/独立审稿 Agent 执行；prompt 中只保留能被下游验证的指令，把"格式类可验证锚点"与"叙事类需人审断言"分离到不同工序，避免模型用格式合规替代叙事质量。
