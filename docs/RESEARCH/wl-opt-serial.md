# 对抗性压测报告：放松 writing-loop 顺序前置、允许同 arc 并行起草

**裁决：survive-with-narrowing（字面命题被证伪；一个大幅收窄的变体成立，但收益远低于宣称的 2-3x）**

被测命题（逐字）：
> episode-writer/reviewer 100% 空跑是因为顺序前置禁止了同 arc 内并行起草。既然节拍单已把每集的「承接：上集末帧」写死了（本就是禁改的 spec），ep-5 可以从 ep-4 的**节拍单**起草而不必等 ep-4 正文 Done。这样能让 2-3 集同时在飞，把一卡包墙钟砍 2-3 倍，同时用 reviewer 邻集对读兜住真断裂。

命题含三个可分别攻击的断言：**(D) 诊断**（空跑=顺序前置所致）、**(M) 机制**（节拍单已把承接写死、够 ground 下一集）、**(P) 收益**（2-3x + 邻集对读兜底）。三条都有致命漏洞，但机制(M) 在收窄后可修复。逐条压。

---

## 0. 一句话结论

命题把**调度层的病（独立定频齐开）误诊为约束层的病（顺序前置）**，并把**节拍单当成了它不是的东西——一个完整冻结的 spec**。真相是：节拍单在系统里被**故意设计成不完整**（§3/§21a 的 EXTRA 收窄「鼓励节拍单未列的创作增量、防填表机器化」），而 ep-5 起草恰恰依赖的两样东西——**实际末帧的重叠帧细节**与 **ep-4 的账本 delta**——都落在这个「故意留白 + 只在 Done 时才落定」的区域里。所以「从节拍单起草」= 用一份**计划/自述**去 ground 下一集，而这正是 §3 明令禁止作为证据的东西（「实现者自述永不作为证据；每个判定输入必须是正文原文或账本事实」）。

它能被收窄到成立，但收窄的代价是**把量产集的节拍单反向改造成完整冻结 spec**——这恰好把系统辛苦躲开的「填表机器化 / citron 式免审执行」请了回来。这笔交易只对**纯量产填充集**划算，对任何承载爽点/钩子手艺的集不划算。

## 1. 攻击 D（诊断）：空跑的主因不是顺序前置，是独立定频齐开——放松顺序前置是**打错靶子**

实测：121 fire / 107 no-op（88%）；episode-writer 29 fire **100%** no-op、reviewer 26 fire **100%** no-op（合计 55 次纯空跑）。命题把这 55 次归因于「顺序前置禁止并行」。这站不住：

- **episode-writer 的 100% 空跑主要是 tier 错配，不是顺序前置。** 当前处于 keystone 阶段：ep1-3 **全归 story-designer 亲写**（keystone tier，§21a §0 拓扑）。episode-writer 的拾取切片里在这个阶段**本就没有属于它 tier 的活**——放不放松顺序前置都一样空跑，因为它能拾的 ep4+ 要么节拍单/大纲门还没放行、要么被 keystone 挡在前面。放松顺序前置只在「ep4-10 子票已放行 Todo、且节拍单已存在」时才可能把空跑转成 fire。
- **reviewer 的 100% 空跑，放松顺序前置几乎救不了。** reviewer 拾取的是 In Review 单集票。并行起草不产生更多 In Review（起草要时间），而且审读**必须按承接序串行**（§21a 审读门 item 3 邻集对读需要 ep-(N-1) 已 Done）。所以 reviewer 的空跑由「没有 In Review 票 + 顶配 reviewer pass 未排」主导，与顺序前置正交。
- **真正的主因是 7 个循环独立定频、全程齐开**（多为 300s）。174 分钟窗口里工作是「串行 + 分阶段」，任一时刻只有 1-2 个 agent 有可拾票 → 其余 5-6 个空跑，每次仍付一次冷启 boot（读 conventions ~41KB + skills + lessons + 板扫描）。这是**调度拓扑病**，不是**约束病**。

**量化归因（粗估）**：55 次纯空跑里，放松顺序前置乐观能转化的只有「episode-writer 在 ep4-10 节拍单已就绪时的那几次」——上限约 7-14 次（7 集量产各一次可拾）；reviewer 侧近乎零转化。其余 ~40+ 次是 phase/cadence/tier 错配，顺序前置一动不动都还在。**即最理想情况下，放松顺序前置只触及 <25% 的空跑浪费，却引入了下面全部的承接返工风险。** 而已提出的方案 A（阶段感知舰队）/ B（cadence 分档）/ D（phase driver 读板判阶段→只起该阶段 agent）直接打掉的是那 ~75%+ 的主因，且零叙事风险。

> **D 裁决：证伪。** 空跑是「独立定频齐开 × 分阶段串行」的调度错配，不是顺序前置。放松顺序前置是**低杠杆、带副作用**的动作；A/B/D 是**高杠杆、零副作用**的动作。先修调度。

## 2. 攻击 M（机制）：节拍单**没有**把 ep-5 依赖的东西写死——「承接」字段是计划，不是落定的正文/账本

这是命题的核心，也是它最脆的地方。逐一拆「节拍单够不够 ground 下一集起草」。

### 2.1 EXTRA 收窄使「实际末帧细节」合法地漂移——重叠帧承接的地基正好在这里

节拍单的承接字段（`templates/arc-beat-card.md` L30）是一句话计划：

```
- 承接：上集末帧 = {…}（重叠帧承接 / 跳时【字幕】声明）
- 尾钩：{H1-H7/H0} {内容}（校验 R1.2/R1.3）
```

而**实际末帧**是 ep-4 正文里的 `▲ 【画面定格】{尾钩画面}`（`templates/episode.md` L26），要求「必须与 frontmatter hook-type 一致」——注意：约束到的是**钩型（H-类）**，不是**具体镜头/具体末句台词/具体调度**。R1.4 更把切点定在「动作势能最高处（按下免提键的一瞬、『你是秦风？』）」——这个**确切的一帧和那句台词是 writer 的手艺**，在设计时（写节拍单时）通常写不死。

关键机制：**§3 与 §21a 把单集票的 EXTRA 判据收窄为「仅违反本集禁写 + 与账本事实冲突」两种**，节拍单模板 L6-7 明文「节拍单未列但不越界的创作增量**合法且鼓励**（防填表机器化）」。也就是说——**ep-4 的 writer 在 EXTRA 允许范围内改动末帧细节，是被系统鼓励的合法行为，不是违规**。

于是链条闭合：ep-5 从节拍单起草时，它的开场重叠帧承接的是节拍单里那句**计划末帧**；ep-4 落地后的**实际末帧**在合法 EXTRA 区里偏移（换了具体镜头/加了一句更狠的末句/调整了定格构图）→ ep-5 的重叠帧承接的是一个**已不存在的帧** → 触发 §19「重叠帧承接」失配 → §21a 审读门 item 3（承接帧接上 ep-(N-1) 末帧）判 fail → 涟漪返工。

**这不是小概率。R1.4/R8.2 恰恰要求 writer 在末帧/尾钩上做最强手艺打磨（金句、切片时刻、动作势能峰值）——末帧是全集手艺密度最高的一帧，也就是最可能被合法 EXTRA 调整的一帧。** 命题说「承接本就写死」，与系统「鼓励在承接附近做未列创作增量」的显式设计**直接矛盾**。

### 2.2 账本 delta 只在 Done 时才落盘——ep-5 起草时看不到 ep-4 的真实状态（citron 教训② 复现）

比末帧更硬的问题在账本。§15 交付义务①：**单集正文 + `ledgers/` 全部更新必须在同一个 commit，工单转态永远在 commit 之后**。即 **ep-4 的 story-state / 信息差表 / 数字锚点 / 伏笔状态的实际 delta，只有在 ep-4 移入 In Review（commit 落盘）后才存在**；ep-5 若从节拍单并行起草，拿到的是**计划的**三轴/信息位阶/伏笔操作（节拍单 L31/L36/L37），不是**实际**账本态。

差在哪？§15 义务②要求 writer 声明**逐条**账本变化并接受「越声明扫描」——这意味着 ep-4 的 writer 在合法 EXTRA 区里做的**任何影响账本的创作增量**（多揭示了一条信息、微调了一个数字锚点、顺手 refresh 了一条伏笔）都会进 story-state 账本，而 ep-5 从节拍单起草时**根本看不到**。后果正是 citron 教训②：「跨集上下文=上一集尾巴，角色状态/关系进展/信息差全靠模型脑补」——ep-5 可能**重复揭示**一个 ep-4 已经揭示的信息（R5.3 Surprise 限量违规 / 信息位阶错乱），或与 ep-4 实际数字锚点冲突（§21a 审读门 item 4 delta 逐条核对判 MISSING/冲突）。

story-state 模板的「逐集末态摘要」（`templates/story-state.md` L35-38）明说它的作用是「使『截至 ep-N 的状态』可重建——修订涟漪分析的依据」。**这行摘要 ep-4 Done 时才写。并行起草的 ep-5 起草于它存在之前。** 这是硬架构约束，不是措辞问题。

### 2.3 因此 M 命题的前提是假的

「节拍单已把承接写死」为真，仅当节拍单是 ep-5 所依赖之物的**完整且冻结的 spec**。但系统**故意**让节拍单不完整（EXTRA 鼓励未列增量）且让**账本 delta 后置到 Done**。ep-5 真正依赖的两样东西——**实际重叠帧** + **ep-4 账本 delta**——恰好都落在「计划≠落定」的缝里。

> **M 裁决：字面证伪。** 用节拍单 ground ep-5 = 用一份计划/自述当证据，正撞 §3「实现者自述永不作为证据；每个判定输入必须是正文原文或账本事实」。ep-4 的正文原文/账本事实在 ep-4 Done 前不存在。

## 3. 攻击「换汤不换药」：躲开了 citron 教训①，却复现了教训②和⑥——而信息差正好落在割裂区

citron #1 死因（autopsy §b-3）：写时看不到上一集正文；批量模式 4/5 集被告知「你是第一集」；无状态账本。对策见 autopsy §f：

- **教训①（无节拍契约）→ 本提案已躲开。** 系统有节拍单（一等 design doc，§21a），citron 完全没有。所以 citron 的「结构性泄洪 EP06 吃掉 EP08 的节拍、EP08 原样重演」这类**结构级**割裂，本系统靠节拍单 + 禁写清单 + 大纲门就挡住了。这一层，本提案**严格优于** citron。
- **教训②（状态账本不可得）→ 本提案原样复现**（见 §2.2）。ep-5 起草时 ep-4 账本 delta 尚未落盘。
- **教训⑥（生成拓扑践踏叙事依赖链）→ 本提案正面违反。** autopsy §f-6 对策**逐字**是：「集 N 开工的前置条件是集 N-1 **已通过验收**（串行链式 + 状态账本传递）；**并行只允许发生在无叙事依赖处（不同项目、同集多稿竞争）**」。本提案把并行开在 ep4→ep5 的**承接依赖边**上——正是对策明令禁止的地方。

**信息差是否正好落在割裂区？是。** citron 的割裂实证是两类：(a) **帧/执行级**——EP05→EP06 笔触/密度肉眼断层（3.6KB 细腻 △ vs 2.5KB 干瘪）；(b) **信息级**——无信息差追踪、重复揭示。节拍单修好的是**结构级**（节拍不重演、不泄洪）；**修不了**的恰是 (a) 实际末帧微调 + (b) 实际账本态——而这两样正是 citron 割裂的两个轴。

更尖锐的一点：命题说「从上一集节拍单起草」。但**节拍单的承接是一句话，比 citron 失败方案里那个 1600-2200 字的正文尾巴还少**。也就是说，在**帧级承接**这条轴上，本提案给 ep-5 的 grounding **比 citron 那个已被判死的方案还弱**。补偿是完整节拍单（结构 spec）——citron 没有的。**净：结构 grounding 更强，帧/账本 grounding 更弱或相当。** 而 citron 死于后者。

> 换汤不换药？**部分是。** 躲开①，复现②⑥。命题担心的「账本状态、实际台词、实际末帧微调」——**确实正好落在导致割裂的那部分**。

## 4. 攻击「量产段真能并行吗」：前向冻结是一个**自动惩罚投机并行**的调速器

命题承认 keystone（ep1-3）本就串行，并行只对量产段有意义。但量产段的并行会被两个既有机制反复打断：

### 4.1 前向冻结（§5 检查2 + §19.5）随失败率把并行拽回串行

§5 检查2：拾取 `Episode: N` 创作票前须验证「不存在 `Episode ≤ N` 的开放 **Bug** 修订票」。§19.5：Bug 修订开放期间，`Episode ≥ 修订集` 的新创作票不可拾取。

在并行量产里，每落地一集，reviewer 邻集对读就可能 file 一张 continuity Bug（§6 邻集复核票，`Episode: N`）。**并行度越高，同一时刻在飞的下游草稿越多，一张前向 Bug 冻结/作废的在飞草稿就越多。** 这是个负反馈调速器：**你越是投机并行（承接假设越可能碎），前向冻结触发越频繁，有效并行度越被拽回串行。** 系统的既有安全机制本身就在抵抗这个提案——强推并行 = 全程和冻结协议对打。

### 4.2 邻集对读是**事后**网，不是**事前**闸；触发即走最贵的路

命题说「用 reviewer 邻集对读兜住真断裂」。但邻集对读（§21a 审读门 item 3 / §19.2 邻集复核票）**只能在 ep-4 Done 之后**把 ep-5 承接帧与 ep-4 **实际末帧**对读——它是**事后诊断**，不是事前预防。它抓到断裂后，修复走 §21a fail 三级路由（notes 回炉 → direct-write → human-park），**这是全系统最贵的路径**。而且：

- 若断裂是**邻集内**（§19.2）：完成修订 + 一张邻集复核票，成本 ≈ 1×(重写 + 复核)。
- 若断裂是**超邻集**（ep-4 实际末帧漂移同时影响并行起草的 ep-5 **和** ep-6）：§19.3 **不得自行开票**，修订票转 `blocked + needs-showrunner`，**showrunner 裁决**——可能升级到**人工停靠**（§19.4 递归上限 ≤2 跳，超限人工停靠）。**并行度越高，一次末帧漂移同时命中多个下游草稿、够到超邻集阈值的概率越高。**

> **人工停靠是墙钟杀手**：它不受算力约束，而是**等操作者**（异步，可能几十分钟到隔夜）。哪怕一次超邻集涟漪触发 showrunner 裁决 → external 停靠，就注入**无界延迟**，抹掉全部并行节省。串行**永不**碰到这个，因为每集写时都对着上一集**实际正文**，承接被 ground、涟漪近乎零。**这是决定性的不对称：串行用吞吐换取零投机返工风险；并行买吞吐但接管了墙钟无界的 human-park 尾险。**

## 5. 量化：一卡包（前 10 集）串行 vs 受控并行墙钟

**建模假设（显式声明）**：
- W = 单集起草成本 ≈ 15-20 min（实测 keystone 出片 ~15-20min），取中值 **17.5 min**。
- V = 单集审读到 Done 成本。**无实测**（reviewer 26 fire 全 no-op，无时长数据）。取两档：**V_cheap = 8 min**（审读门=引文核对清单）/ **V_heavy = 15 min**（顶配 reviewer 逐条引文）。
- 一卡包=前 10 集；ep1-3 keystone（story-designer 亲写，串行不可约）；ep4-10 = 7 量产集。
- 顺序前置下 In Review 计「开放」（§5 检查1 逐字），故 ep-N 须等 ep-(N-1) **Done** 才开工 → 串行 = 逐集 (W+V) 首尾相接。
- p = 并行起草的量产集需承接返工的概率。**无实测**（项目从未到量产段）——这是全模型最不确定的输入。
- 返工成本 R ≈ (W+V)/集（邻集内）；超邻集 → +human-park（保守记 60 min 一次操作者延迟，实际可到隔夜）。

### 串行基线
临界路径 = 10 × (W+V)（严格串行；ep-N 等 ep-(N-1) Done）。
- V_cheap：10×25.5 = **255 min ≈ 4.25 hr**
- V_heavy：10×32.5 = **325 min ≈ 5.4 hr**

### 受控并行（3 集在飞）
量产 7 集的**起草**可在节拍单就绪后全部并发启动（这正是提案的机制），瓶颈从「串行起草链」移到「串行审读链」（审读必须按承接序、且需前集 Done）。
临界路径 ≈ W_首集 + 10×V（串行审读链，起草被藏在审读链下）+ 返工 + human-park 尾险。

| 情形 | T_parallel | 相对串行加速 |
|---|---|---|
| **零返工上限** p=0, V_cheap | 17.5 + 80 = 97.5 min | **2.6x** |
| 零返工上限 p=0, V_heavy | 17.5 + 150 = 167.5 min | **1.94x** |
| 现实 p=0.25, V_cheap | 97.5 + 0.25×7×25.5 = 142 min | **1.8x** |
| 现实 p=0.4, V_heavy | 167.5 + 0.4×7×32.5 = 258 min | **1.26x** |
| p=0.4 + 一次超邻集 human-park(+60) | 318 min | **1.02x（≈打平）** |
| p=0.4 + human-park 隔夜(+数小时) | > 串行 | **< 1x（并行更慢）** |

**读法**：
1. **命题的 2-3x 只在零返工极限成立**，且原因是「消掉串行起草链、只剩串行审读链」——数学上真的能到 2-2.6x（前提 V_cheap，审读远便宜于起草）。
2. **一旦返工进场（p=0.25-0.4），实际加速塌到 1.3-1.8x**，远非 2-3x。
3. **任何一次超邻集 → human-park 事件，加速塌到 ≤1x，可为负。** 决定性变量是 p 和超邻集尾险，而**我们对 p 零实测**（项目从未到量产段），且系统自身先验是「AI 初稿 fail 是常态不是事故」（§21a fail 三级路由的存在理由）。**在系统自己的先验下，赌 p 低是没有依据的。**

一个常被忽略的成本：**并行不并行审读**。审读串行 + 需前集 Done，且 reviewer 是 ≥writer 档、keystone 需顶配（§0/§21a）——**并行只是把瓶颈从便宜模型（sonnet 起草）挪到贵模型（opus 审读），总墙钟未必动多少，token 成本还上升。**

---

## 6. 若要收窄到成立：精确安全条件（每一条都对应上面一处攻击）

命题可被收窄到成立，但收窄条件把 p 压到近零、把尾险封顶，代价是牺牲量产集创作自由：

1. **仅量产段，且排除 keystone 邻接集。** 只对 tier=episode-writer 且**其承接的前集也是量产集**的集并行。承接 keystone 末帧的集（如 ep-4 承接 ep-3 keystone）**不并行**——keystone 末帧 = 最强钩 = 最可能在 EXTRA 区被手艺打磨的一帧。
2. **给节拍单加「承接帧冻结」字段并设为禁改。** 承接的重叠帧（确切调度 + 末句台词）必须在节拍单里写成**确切帧**，并对量产集 writer **列入本集禁写等价的冻结项**——动它 = EXTRA fail。**代价**：这与节拍单 L6-7「防填表机器化、鼓励未列增量」直接冲突，也与 R1.4「切点在动作势能最高处（是执行时手艺）」冲突——你在要求设计时就把最难的一帧写死。
3. **冻结量产集的账本 delta。** 量产集的计划三轴/信息位阶/伏笔操作被当作**完整**账本 delta；任何超出节拍单的**影响账本**的创作增量 = EXTRA fail。**代价**：撞 R6.1（每集须推进 ≥2 轴，含信息/闭环轴——这些**就是**账本操作；全预定=writer 轴推进自由归零）+ 再次撞「防填表机器化」。**这一条本质上把量产节拍单变成完整冻结 spec——把系统躲开的 citron 式「照单机械执行」请回来。**
4. **仅在无开放前向 Bug 的窗口启动并行；一旦前向 Bug 落地，所有在飞下游草稿立即作废重校**（不是冻结后原样复用——它们的前提集将改）。
5. **邻集对读从审读门清单项升级为强制阻断门**：任何并行起草集在其邻集对读（对**实际前集正文**，不是节拍单）通过前不得 Done；fail → 强制回炉，禁橡皮图章。
6. **超邻集涟漪 = 立即人工停靠**（§19.3/19.4），接受「这一批并行节省被这次事件吃掉」。并行度 ≤ 2-3，且被「距下一 keystone / 下一卡点门的集数」封顶——绝不并行越过里程碑门（evaluator 门本就以 Blocked-by 挡住，§21）。

**在 1-6 下**，p 被压低、尾险封顶，量产段起草相能拿到 ~1.5-2x（§5 表）。但注意：条件 2+3 是一次**真实的 doctrine 变更**（从 EXTRA 鼓励创作 → 量产集机械执行冻结卡）。这笔账**只对纯量产填充集划算**；对任何承载爽点/钩子/金句的集，牺牲的正是它的价值来源。

## 7. 哪条铁律不能让（即便在收窄变体里也必须 hold）

- **§15 单 commit 原子性 + 账本 delta 后置到 Done + §3「只有正文原文/账本事实是证据」**：下游集不得消费**未落盘**的账本态。这正是为什么收窄变体必须把账本 delta 预冻结进卡（条件3），否则并行不安全。**这是字面命题违反、且不能豁免的那条铁律**——「从节拍单（计划/自述）起草」= 用非证据当地基。
- **§5 检查2 / §19.5 前向冻结**：并行草稿绝不得越过一张开放前向 Bug 到达 Done——那是账本事实冲突传播的通道。
- **§21a 审读门 item 3 邻集对读须对实际前集正文、且在前集 Done 之后**：不得用「对节拍单对读」替代（对读节拍单只能验证承接**计划**，验不了实际帧）。且须升级为阻断门。
- **§19.3 超邻集 → showrunner 裁决**：不得自行批量开返工票。
- **keystone 串行 + 顶配 reviewer floor（§0 拓扑 / §21a）**：keystone 不得从前集卡并行起草。

## 8. 落地建议(排序)

1. **先修调度，别动约束。** 空跑的 ~75%+ 是独立定频齐开 × 分阶段串行（§1）。上方案 D（phase driver 读板判阶段 → 只起该阶段 agent）+ 方案 B（cadence 分档），零叙事风险打掉主因。放松顺序前置**最多触及 <25% 空跑、却引入承接返工全谱风险**——错靶。
2. **无论并行与否，把邻集对读升级为强制阻断门**（对实际前集正文）。这是独立的净收益：即便纯串行，它也把当前「审读门清单项」这一较弱的承接检查坐实。低成本。
3. **并行只在收窄封套内试点，且先测 p。** 在赌并行之前，先让**一个量产 arc 走纯串行**，实测 reviewer 邻集对读 fail 率 = p 的经验估计。p 是全模型最不确定、也最决定性的输入，而我们现在**零数据**（项目从未到量产）。拿不到 p 就上并行 = 拿墙钟去赌一个系统先验说「常态会 fail」的量。
4. **若确要并行**：仅量产、冻结承接帧+账本 delta 进卡（接受 doctrine 变更与 R6.1/防填表机器化的冲突）、前向冻结窗口内启动、并行度 ≤3、超邻集即 human-park。收益按 §5 表约 1.3-1.8x（非 2-3x），且仅在量产填充段。
